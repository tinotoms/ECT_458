IEEE 802.15.4 — Physical Layer ★★★ Every Paper

Standard for Low-Rate Wireless PANs (LR-WPANs). Used by ZigBee, 6LoWPAN.

Frequency Bands & Data Rates:

  • 2.4 GHz — Worldwide, 16 channels, 250 kbps
  • 868 MHz — Europe, 1 channel, 20 kbps
  • 915 MHz — Americas, 10 channels, 40 kbps

Modulation Techniques:

  • BPSK PHY — Direct Sequence Spread Spectrum (DSSS); 2 phase shifts
  • OQPSK PHY — Offset QPSK; 4 phase shifts; offset improves reliability
  • ASK PHY — Parallel Sequence Spread Spectrum (PSSS); better range & throughput

PHY Frame Format:

Preamble (4B)SFD (1B)PHR (1B)PHY Payload (0–127B)FCS (2B)
Sync/timingEnd of preambleData lengthContains MAC frameError detection

Max frame = 127 bytes. Channel access = CSMA-CA.

IEEE 802.15.4 — MAC Layer & Security ★★★

MAC Responsibilities:

  • Channel access via CSMA-CA (Carrier Sense Multiple Access with Collision Avoidance)
  • Network beaconing (coordinator broadcasts beacon for synchronization)
  • PAN association/disassociation, reliable link comms via ACK frames

MAC Frame Types:

  • Data Frame — Carries application/network layer data
  • Beacon Frame — PAN coordinator → synchronization & network timing
  • Acknowledgement Frame — Confirms successful reception
  • MAC Command Frame — Association, disassociation, management

MAC General Frame Format:

Frame Control (2B)Seq No. (1B)Addr Fields (var)MAC Payload (var)FCS (2B)
Frame type + flagsDeduplicationSrc/Dst PAN ID + addrData/beacon/cmdIntegrity check

Security (AES-128 at MAC layer):

  • AES-128 Encryption — Base algorithm for all security modes
  • MIC (Message Integrity Code) — Computed using AES over entire frame; receiver recomputes and compares
  • Auxiliary Security Header — Added when security enabled: Security Control + Frame Counter + Key Identifier
  • AES-CCM* — Supports: encryption only, MIC only, or both combined

Security-enabled frame: MHR | Aux Security Header | Encrypted Payload | MIC (4/8/16B) | FCS

802.15.4e — TSCH explained:

Combines TDMA + FDMA. Time divided into fixed slots; nodes transmit on different channels at different times → reduces interference & multipath fading.

802.15.4g — New PHY Modes:

  • MR-FSK — Constant envelope signal, good transmit power efficiency
  • MR-OFDM — Higher data rates; may be too complex for ultra-low-cost devices
  • MR-OQPSK — Makes multi-mode systems more cost-effective

LoRaWAN — Architecture & Physical Layer ★★★ Every Paper

LPWAN protocol for long-range battery-operated IoT. LoRa PHY by Semtech. MAC/architecture by LoRa Alliance.

Star-of-Stars Architecture:

ComponentRole
End Devices (Endpoints)Battery-powered IoT sensors. Communicate wirelessly to one or more gateways using LoRa modulation
LoRa GatewaysMulti-channel transparent bridges. Relay LoRa packets to network server via IP (Ethernet/4G). Can handle multiple SFs simultaneously
Network ServerCentral controller: MAC commands, deduplication (from multi-gateways), ADR (Adaptive Data Rate), routing to app server
Application ServerDecrypts application payloads; provides data to IoT cloud platforms

Physical Layer — Chirp Spread Spectrum (CSS):

  • Each bit spread using a chirp signal. Number of chips/bit = Spreading Factor (SF7–SF12)
  • Higher SF → longer range, higher sensitivity, lower data rate, more airtime
  • Can demodulate signals below the noise floor
  • Frequency bands: 433 MHz, 868 MHz (EU), 915 MHz (US) — sub-GHz unlicensed
SFData Rate (125kHz BW)Range
SF75470 bpsShortest
SF91760 bpsMedium
SF12250 bpsLongest (~15 km rural)

ADR (Adaptive Data Rate) — Endpoints close to gateway use high data rate & low power; distant endpoints use low data rate & high power.

6LoWPAN Adaptation Layer ★★★ Every Paper

IPv6 over Low-Power Wireless PANs. IETF adaptation layer between IEEE 802.15.4 and IPv6. RFCs 4944, 6282, 6775. Only supports IPv6 (not IPv4).

Core Problem: IPv6 min MTU = 1280 bytes; 802.15.4 max frame = 127 bytes. IPv6 header = 40B + UDP = 8B — too large!

Three Key Functions:

1. Header Compression (IPHC — RFC 6282):

  • Compresses IPv6 (40B) + UDP (8B) headers down to as few as 6 bytes combined
  • Elides fields with assumed values: Version, Traffic Class, Flow Label, Payload Length, Next Header, Hop Limit, Src/Dst Addresses (derived from IEEE MAC addresses)

2. Fragmentation & Reassembly:

  • Large IPv6 packets fragmented into multiple 127-byte 802.15.4 frames at Layer 2
  • Datagram Size (11 bits) — Total unfragmented payload size
  • Datagram Tag (16 bits) — Identifies all fragments of same packet
  • Datagram Offset (8 bits) — Where this fragment starts in original payload (multiples of 8)
  • First fragment = FRAG1 (no offset); subsequent = FRAGN (with offset)

3. Mesh Addressing:

  • Layer-2 multi-hop packet forwarding without full IP stack at each intermediate node
  • Fields: Hop Limit, Source MAC address, Destination MAC address

Dispatch byte at start of frame indicates active capabilities: NALP / Uncompressed IPv6 / IPHC / Mesh / FRAG1 / FRAGN

RPL — Routing Protocol for Low Power and Lossy Networks ★★★

IPv6 distance-vector routing protocol (IETF RFC 6550) for IoT. Works in Low Power and Lossy Networks (LLNs) with high packet loss, limited memory/CPU, battery constraints.

Network Structure — DODAG:

Builds a Destination-Oriented Directed Acyclic Graph (DODAG) rooted at a border router. Each node maintains up to 3 parents (preferred parent = best next-hop toward root). Rank approximates distance from root, preventing routing loops.

Route Discovery Messages:

MessageDirectionPurpose
DIO (DAG Information Object)Downward from rootNodes discover parents, calculate rank → build upward routes
DAO (Destination Advertisement Object)Upward to rootInform parents of reachability → establish downward routes
DIS (DAG Information Solicitation)New node broadcastsRequest DIO to join DODAG

Two Memory Modes:

  • Storing Mode — All nodes keep full routing tables. Optimal paths. Higher memory usage.
  • Non-Storing Mode — Only border router holds table; source-routes downward. Memory-efficient.

Objective Function & Metrics:

Defines how metrics combine to compute rank. Metrics: ETX (Expected Transmission Count), Hop Count, Latency, Link Quality Level, Node Energy, Throughput, Link Color.

Modbus Protocol ★★

Industrial serial communication protocol by Modicon (1979). Widely used in industrial IoT, SCADA, building automation.

Key Features:

  • Master-Slave Architecture — Communication always initiated by master (HMI/SCADA); slaves (sensors, PLCs) only respond when queried
  • Request-Response Protocol — Every communication = one request + one response
  • Open Standard — Freely available, widely supported
  • Serial & TCP/IP Support — Runs over RS-232, RS-485 serial lines or Ethernet (Modbus TCP)
  • Transmission Modes — Modbus RTU (binary, compact), Modbus ASCII (text), Modbus TCP (Ethernet)
  • Register-Based Data Model — Coils (1-bit), Discrete inputs, Holding registers, Input registers
  • Applications — SCADA, building automation, energy meters, industrial IoT gateways

Need for IP Optimization in IoT & Next-Gen IP Protocols ★★

Why IP Optimization is Needed:

  • IoT devices have very limited RAM and CPU — standard IP stacks too large
  • 802.15.4 max frame = 127 bytes; IPv6 min MTU = 1280 bytes (huge mismatch)
  • Large headers waste battery — compression reduces TX time and power
  • Wireless IoT links are lossy — routing must handle high packet loss (RPL)
  • Billions of devices need IPv6 128-bit addressing, but full headers impractical

GPIO Programming — LED & Push Button (Python) ★★★ Every Paper

Message Integrity & Non-Repudiation ★★★

Message Integrity: Ensures data is unaltered during transmission/storage.

Hash/Digest Process:

  • Step 1: Sender computes h0 = Hash(M0, K) using secret key K
  • Step 2: h0 appended to message and sent/stored
  • Step 3: Receiver computes h1 = Hash(M1, K)
  • Step 4: h1 = h0 → PASS (unchanged); h1 ≠ h0 → FAIL (tampered/corrupted)
MD5 → 128-bit digest | SHA-256 → 256-bit digest | AES-128 MIC → used in 802.15.4 & LoRaWAN

Non-Repudiation: Digital signature certified by Trusted Third Party (TTP). Sender cannot later deny having sent the message.

In LoRaWAN: NwkSKey computes MIC at MAC layer; AppSKey secures the payload.

M3: IEEE 802.15.4 PHY + MAC + Security ★★★

PHY: 2.4GHz/250kbps, 868MHz/20kbps, 915MHz/40kbps. Modulation: BPSK (DSSS), OQPSK, ASK (PSSS). Frame: Preamble(4B)+SFD(1B)+PHR(1B)+Payload(0-127B)+FCS(2B). MAC: CSMA-CA, 4 frame types (Data/Beacon/ACK/Command). Frame: FrameCtrl(2B)+SeqNo(1B)+Addr(var)+Payload+FCS(2B). Security: AES-128, MIC computed over entire frame, AES-CCM* modes, Auxiliary Security Header (SecurityCtrl+FrameCounter+KeyID).

M3: IEEE 802.15.4e vs 802.15.4g ★★★

802.15.4e = MAC layer amendment. TSCH (TDMA+FDMA, time slots, channel hopping), IEs (Header+Payload), Enhanced Beacons, Enhanced ACK. Target: factory automation, smart grid. Payload still 127 bytes. 802.15.4g = PHY layer amendment. New PHYs: MR-FSK/MR-OFDM/MR-OQPSK. Payload increased to 2047 bytes. Frequency 169MHz–2.4GHz. Target: smart utility networks, smart grid outdoor.

M3: ZigBee Stack Architecture ★★★

Stack (top-down): APL (Application Layer + ZDO) → APS (binding, group mgmt) → NWK (AODV routing, AES-128 security) → 802.15.4 MAC → 802.15.4 PHY. Device types: Coordinator (ZC, 1 per network, forms PAN), Router (ZR, FFD, extends coverage), End Device (ZED, RFD, low power, talks only to parent). Features: 65000 nodes, AES-128 dual-layer security, self-healing mesh, AODV on-demand routing, topologies: Star/Tree/Mesh.

M3: LoRaWAN Architecture + PHY + MAC ★★★

Architecture: End Devices → LoRa Gateways (multi-channel transparent bridge) → Network Server (dedup, ADR, MAC cmds) → Application Server. Star-of-Stars topology. PHY: CSS (Chirp Spread Spectrum), SF7–SF12 (higher SF = longer range, lower rate). SF12=250bps, SF7=5470bps. Sub-GHz unlicensed bands. Device classes: A (2 RX windows after uplink, lowest power), B (scheduled via beacon), C (always listening, mains-powered). Security: NwkSKey (MAC/MIC) + AppSKey (payload encryption). DevAddr=32-bit (NwkID 7bits + NwkAddr 25bits).

M3: LTE-M vs NB-IoT ★★★

Both 3GPP Release 13. LTE-M: 1.4MHz BW, 200kbps, voice support (VoLTE), full mobility, in-band LTE only, +15dB coverage, ~10yr battery. NB-IoT: 200kHz BW, 60kbps UL/30kbps DL, no voice, limited mobility, 3 deployment modes (Standalone/In-band/Guard Band), +20dB vs GPRS, >10yr battery. Both support PSM + eDRX. NB-IoT standalone mode reuses GSM 200kHz channel — ideal for 2G migration.

M3: 6LoWPAN Adaptation Layer ★★★

IPv6 over 802.15.4 (IETF RFC 4944, 6282). Problem: IPv6 MTU=1280B, 802.15.4 max=127B; IPv6 header=40B+UDP=8B. Three functions: (1) Header Compression (IPHC): compresses to 6 bytes — elides version, traffic class, flow label, addresses from MAC context. (2) Fragmentation: Datagram Size(11b)+Tag(16b)+Offset(8b); FRAG1 (no offset) then FRAGN (with offset). (3) Mesh Addressing: Layer-2 multi-hop via Hop Limit+Src+Dst MAC fields. Dispatch byte indicates active caps.

M3: RPL Routing Protocol ★★★

RFC 6550. IPv6 distance-vector for Low Power Lossy Networks (LLNs). Builds DODAG (Destination-Oriented DAG) rooted at border router. Messages: DIO (downward from root → build upward routes), DAO (upward → build downward routes), DIS (new node requests DIO). Rank = distance from root, prevents loops. Two modes: Storing (all nodes keep routing table, memory-heavy) vs Non-Storing (only border router, memory-efficient, source-routed). Metrics: ETX, Hop Count, Latency, LQL, Node Energy, Throughput, Link Color.

M4: Xively + Nimbits Platforms ★★★

Xively: PaaS (formerly Cosm/Pachube, now Google). First public cloud for IoT. 4 concepts: Data Stream (continuous flow), Data Points (individual values), Triggers (actions on state change), Feed (set of streams/alerts). Pull=auto HTTP server; Push=manual HTTP client. Protocols: REST/MQTT/WebSockets. Formats: JSON/XML/CSV. Nimbits: PaaS, open-source. Runs on Google App Engine OR RPi. Components: NimbitsServerL (calculation objects), XMPPServerL (alerts), XMPP protocol, H2 Database, Security Tokens. Edge computing, rule engine, XMPP alerts, nimbits.io Java library.

M4: Raspberry Pi + UART/SPI/I2C ★★★ Every Paper

RPi: ARM Broadcom SoC (700MHz, 512MB SDRAM Rev2), 40-pin GPIO, USB×4, HDMI+RCA, MicroSD (min 8GB). UART=2-wire (TX-RX crossed), async, 115200bps, full-duplex, 1:1. SPI=4-wire (MOSI/MISO/SCLK/SS), sync, full-duplex, 8Mbps+, one SS per slave. I2C=2-wire (SDA/SCL+pull-ups), sync, half-duplex, 400kbps, unique 7/10-bit address, multi-device same bus. Memory hook: UART=simple/slow/1:1; SPI=fast/more wires/no address; I2C=2-wire/addressable/multi-device.

M4: GPIO LED + Button Code ★★★ Every Paper

GPIO.setmode(GPIO.BCM) → setup(20, OUT) → setup(21, IN, PUD_UP). If GPIO.input(21)==False: button pressed (active LOW), output(20, HIGH) = LED ON. Else output(20, LOW) = LED OFF. LED: GPIO20 → 330Ω → LED(+) → LED(−) → GND. Button: GPIO21 ↔ pushbutton ↔ GND. Run: sudo python led.py. Always GPIO.cleanup() in finally block. BCM = Broadcom numbering; BOARD = physical pin numbering.

FC1=Identity Mgmt (IdM), FC2=Authentication (MD5/SHA hash, certs — only hash transmitted), FC3=Authorisation (ACL=coarse/RBAC=role-based/ABAC=attribute-finest), FC4=Key Mgmt (gen/distrib/store/revoke), FC5=Trust & Reputation (dynamic trust scores). M5: Layered Attacker Model + Smart City Architecture ★★★ Every Paper

L1 Physical: eavesdrop/jam/tamper → AES-CCM 128. L2 Data Link: MAC spoof/ARP poison/VLAN hop → ARP inspection, port security.

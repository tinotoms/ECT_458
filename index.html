IEEE 802.15.4 — Physical Layer ★★★ Every Paper5×

Standard for Low-Rate Wireless PANs (LR-WPANs). Used by ZigBee, 6LoWPAN.

Frequency Bands & Data Rates:

  • 2.4 GHz — Worldwide, 16 channels, 250 kbps
  • 868 MHz — Europe, 1 channel, 20 kbps
  • 915 MHz — Americas, 10 channels, 40 kbps

Modulation Techniques:

  • BPSK PHY — Direct Sequence Spread Spectrum (DSSS); 2 phase shifts
  • OQPSK PHY — Offset QPSK; 4 phase shifts; offset improves reliability
  • ASK PHY — Parallel Sequence Spread Spectrum (PSSS); better range & throughput

PHY Frame Format:

Preamble (4B)SFD (1B)PHR (1B)PHY Payload (0–127B)FCS (2B)
Sync/timingEnd of preambleData lengthContains MAC frameError detection

Max frame = 127 bytes. Channel access = CSMA-CA.

IEEE 802.15.4 — MAC Layer & Security ★★★5×

MAC Responsibilities:

  • Channel access via CSMA-CA
  • Network beaconing (coordinator broadcasts beacon for synchronization)
  • PAN association/disassociation, reliable link comms via ACK frames

MAC Frame Types:

  • Data Frame — Carries application/network layer data
  • Beacon Frame — PAN coordinator → synchronization & network timing
  • Acknowledgement Frame — Confirms successful reception
  • MAC Command Frame — Association, disassociation, management

MAC General Frame Format:

Frame Control (2B)Seq No. (1B)Addr Fields (var)MAC Payload (var)FCS (2B)
Frame type + flagsDeduplicationSrc/Dst PAN ID + addrData/beacon/cmdIntegrity check

Security (AES-128 at MAC layer):

  • AES-128 Encryption — Base algorithm for all security modes
  • MIC (Message Integrity Code) — Computed using AES over entire frame; receiver recomputes and compares
  • Auxiliary Security Header — Security Control + Frame Counter + Key Identifier
  • AES-CCM* — Supports: encryption only, MIC only, or both combined

Security-enabled frame: MHR | Aux Security Header | Encrypted Payload | MIC (4/8/16B) | FCS

802.15.4e — TSCH explained:

Combines TDMA + FDMA. Time divided into fixed slots; nodes transmit on different channels at different times → reduces interference & multipath fading.

802.15.4g — New PHY Modes:

  • MR-FSK — Constant envelope signal, good transmit power efficiency
  • MR-OFDM — Higher data rates; may be too complex for ultra-low-cost devices
  • MR-OQPSK — Makes multi-mode systems more cost-effective

  • ZigBee Router (ZR) — Full-function device (FFD); extends coverage, participates in routing
  • ZigBee End Device (ZED) — Reduced-function device (RFD); communicates only with parent; optimized for low power

  • AODV Routing — On-demand; route discovered only when needed → minimizes overhead

  • Self-forming and self-healing mesh networks

LoRaWAN — Architecture & Physical Layer ★★★ Every Paper5×

LPWAN protocol for long-range battery-operated IoT. LoRa PHY by Semtech. MAC/architecture by LoRa Alliance.

Star-of-Stars Architecture:

ComponentRole
End Devices (Endpoints)Battery-powered IoT sensors. Communicate wirelessly to one or more gateways using LoRa modulation
LoRa GatewaysMulti-channel transparent bridges. Relay LoRa packets to network server via IP (Ethernet/4G). Can handle multiple SFs simultaneously
Network ServerCentral controller: MAC commands, deduplication (from multi-gateways), ADR (Adaptive Data Rate), routing to app server
Application ServerDecrypts application payloads; provides data to IoT cloud platforms

Physical Layer — Chirp Spread Spectrum (CSS):

  • Each bit spread using a chirp signal. Number of chips/bit = Spreading Factor (SF7–SF12)
  • Higher SF → longer range, higher sensitivity, lower data rate, more airtime
  • Can demodulate signals below the noise floor
  • Frequency bands: 433 MHz, 868 MHz (EU), 915 MHz (US) — sub-GHz unlicensed
SFData Rate (125kHz BW)Range
SF75470 bpsShortest
SF91760 bpsMedium
SF12250 bpsLongest (~15 km rural)

ADR (Adaptive Data Rate) — Endpoints close to gateway: high data rate & low power. Distant: low data rate & high power.

LoRaWAN Security (Two-Layer AES-128):

  • NwkSKey — MAC layer: endpoint authentication + MIC computation/verification
  • AppSKey — Encrypts/decrypts app payloads. Network server cannot read app data

  • AppEUI — Global application provider ID (IEEE EUI-64)
  • DevAddr (32-bit) — MSBs (7 bits) = NwkID; LSBs (25 bits) = NwkAddr

6LoWPAN Adaptation Layer ★★★ Every Paper3×

IPv6 over Low-Power Wireless PANs. IETF RFCs 4944, 6282, 6775. Only supports IPv6 (not IPv4).

Core Problem: IPv6 min MTU = 1280 bytes; 802.15.4 max frame = 127 bytes. IPv6 header = 40B + UDP = 8B — too large!

Three Key Functions:

1. Header Compression (IPHC — RFC 6282):

  • Compresses IPv6 (40B) + UDP (8B) headers down to as few as 6 bytes combined
  • Elides fields with assumed values: Version, Traffic Class, Flow Label, Payload Length, Next Header, Hop Limit, Src/Dst Addresses

2. Fragmentation & Reassembly:

  • Large IPv6 packets fragmented into multiple 127-byte 802.15.4 frames at Layer 2
  • Datagram Size (11 bits) — Total unfragmented payload size
  • Datagram Tag (16 bits) — Identifies all fragments of same packet
  • Datagram Offset (8 bits) — Where this fragment starts (multiples of 8)
  • First fragment = FRAG1 (no offset); subsequent = FRAGN (with offset)

3. Mesh Addressing:

  • Layer-2 multi-hop packet forwarding without full IP stack at each node
  • Fields: Hop Limit, Source MAC address, Destination MAC address

Dispatch byte at start of frame indicates active capabilities: NALP / Uncompressed IPv6 / IPHC / Mesh / FRAG1 / FRAGN

RPL — Routing Protocol for Low Power and Lossy Networks ★★★3×

IPv6 distance-vector routing protocol (IETF RFC 6550) for IoT.

Network Structure — DODAG:

Builds a Destination-Oriented Directed Acyclic Graph (DODAG) rooted at a border router. Each node maintains up to 3 parents. Rank approximates distance from root, preventing routing loops.

Route Discovery Messages:

MessageDirectionPurpose
DIO (DAG Information Object)Downward from rootNodes discover parents, calculate rank → build upward routes
DAO (Destination Advertisement Object)Upward to rootInform parents of reachability → establish downward routes
DIS (DAG Information Solicitation)New node broadcastsRequest DIO to join DODAG

Two Memory Modes:

  • Storing Mode — All nodes keep full routing tables. Optimal paths. Higher memory usage.
  • Non-Storing Mode — Only border router holds table; source-routes downward. Memory-efficient.

Objective Function & Metrics:

Metrics: ETX (Expected Transmission Count), Hop Count, Latency, Link Quality Level, Node Energy, Throughput, Link Color.

Modbus Protocol ★★3×

Industrial serial communication protocol by Modicon (1979). Used in SCADA, building automation, industrial IoT.

Key Features:

  • Master-Slave Architecture — Communication always initiated by master (HMI/SCADA)
  • Request-Response Protocol — Every communication = one request + one response
  • Open Standard — Freely available, widely supported
  • Serial & TCP/IP Support — RS-232, RS-485 or Ethernet (Modbus TCP)
  • Transmission Modes — Modbus RTU (binary), Modbus ASCII (text), Modbus TCP (Ethernet)
  • Register-Based Data Model — Coils (1-bit), Discrete inputs, Holding registers, Input registers

Need for IP Optimization & Next-Gen IP Protocols ★★2×

Why IP Optimization is Needed:

  • IoT devices have very limited RAM and CPU — standard IP stacks too large
  • 802.15.4 max frame = 127 bytes; IPv6 min MTU = 1280 bytes (huge mismatch)
  • Large headers waste battery — compression reduces TX time and power
  • Wireless IoT links are lossy — routing must handle high packet loss (RPL)
  • Billions of devices need IPv6 128-bit addressing, but full headers impractical

GPIO Programming — LED & Push Button (Python) ★★★ Every Paper2×

import RPi.GPIO as GPIO import time GPIO.setmode(GPIO.BCM) # BCM Broadcom pin numbering GPIO.setup(20, GPIO.OUT) # GPIO 20 = LED output GPIO.setup(21, GPIO.IN, pull_up_down=GPIO.PUD_UP) # GPIO 21 = Button + pull-up try: while True: if GPIO.input(21) == False: # Button pressed → GPIO reads LOW GPIO.output(20, GPIO.HIGH) # LED ON (3.3V) else: GPIO.output(20, GPIO.LOW) # LED OFF (0V) time.sleep(0.1) # 100ms polling finally: GPIO.cleanup() # Reset all pins on exit
  • LED circuit: GPIO20 → 330Ω resistor → LED Anode(+) → LED Cathode(−) → GND
  • Button circuit: GPIO21 ↔ Push button ↔ GND | Pull-up → reads HIGH when open, LOW when pressed

Message Integrity & Non-Repudiation ★★★5×

Message Integrity: Ensures data is unaltered during transmission/storage.

Hash/Digest Process:

  • Step 1: Sender computes h0 = Hash(M0, K) using secret key K
  • Step 2: h0 appended to message and sent/stored
  • Step 3: Receiver computes h1 = Hash(M1, K)
  • Step 4: h1 = h0 → PASS (unchanged); h1 ≠ h0 → FAIL (tampered/corrupted)
MD5 → 128-bit digest | SHA-256 → 256-bit digest | AES-128 MIC → used in 802.15.4 & LoRaWAN

Non-Repudiation: Digital signature certified by Trusted Third Party (TTP). Sender cannot deny having sent the message.

In LoRaWAN: NwkSKey computes MIC at MAC layer; AppSKey secures the payload.

M3: IEEE 802.15.4 PHY + MAC + Security 5×

PHY: 2.4GHz/250kbps, 868MHz/20kbps, 915MHz/40kbps. Modulation: BPSK (DSSS), OQPSK, ASK (PSSS). Frame: Preamble(4B)+SFD(1B)+PHR(1B)+Payload(0-127B)+FCS(2B). MAC: CSMA-CA, 4 frame types (Data/Beacon/ACK/Command). Frame: FrameCtrl(2B)+SeqNo(1B)+Addr(var)+Payload+FCS(2B). Security: AES-128, MIC computed over entire frame, AES-CCM* modes, Auxiliary Security Header (SecurityCtrl+FrameCounter+KeyID).

M3: LoRaWAN Architecture + PHY + MAC 5×

Architecture: End Devices → LoRa Gateways (multi-channel transparent bridge) → Network Server (dedup, ADR, MAC cmds) → Application Server. Star-of-Stars topology. PHY: CSS (Chirp Spread Spectrum), SF7–SF12 (higher SF = longer range, lower rate). SF12=250bps, SF7=5470bps. Sub-GHz unlicensed bands. Device classes: A (2 RX windows after uplink, lowest power), B (scheduled via beacon), C (always listening, mains-powered). Security: NwkSKey (MAC/MIC) + AppSKey (payload encryption). DevAddr=32-bit (NwkID 7bits + NwkAddr 25bits).

M5: Message Integrity 5×

h0 = Hash(M0, K) at sender → transmitted with message → receiver computes h1 = Hash(M1, K) → h1==h0 PASS else FAIL. Algorithms: MD5 (128-bit), SHA-256 (256-bit). In 802.15.4/LoRaWAN: AES-128 MIC. Non-repudiation: digital signature certified by TTP → sender cannot deny.

M5: 4-Layer Smart City Architecture 5×

L1 Street: sensors (parking/air/cameras) + ZigBee/LoRaWAN gateways → L2 City Network: routers + REP (≥2 paths to DC) → L3 Data Center: cloud + fog edge processing → L4 Services: city apps + citizen apps + APIs. REP = Resilient Ethernet Protocol. Fog = real-time edge decisions. Cross-domain: humidity data adjusts street light intensity.

M4: XaaS Cloud Service Models 5×

SaaS: software on demand (Gmail, Drive) — user controls nothing. PaaS: platform for devs (App Engine, Xively, Nimbits, Azure) — user controls applications. IaaS: infra on rent (AWS, Azure, GCE) — user controls apps+OS+data. DaaS: data warehouse/virtual desktop (iCloud, Tata 10X) — user controls data. Provider manages everything below each model's control line.

🟡 HIGH — appeared 3×

Quick-fire 3× answers

  • RPL: RFC 6550. DODAG rooted at border router. DIO (root downward → upward routes), DAO (upward → downward routes), DIS (new node → request DIO). Rank prevents loops. Storing (all nodes keep table) vs Non-Storing (only border router, source-routed).
  • 6LoWPAN: IPv6 over 802.15.4 (RFC 4944/6282). Three functions: (1) IPHC compression (40B+8B → 6B), (2) Fragmentation (FRAG1 + FRAGN, fields: Size 11b + Tag 16b + Offset 8b), (3) Mesh addressing (Hop Limit + Src/Dst MAC). Dispatch byte signals active capability.
  • NB-IoT network layer: OFDMA DL (15kHz sub-carrier), SC-FDMA UL (15/3.75kHz), 200kHz BW, half-duplex FDD, 60/30kbps, +20dB vs GPRS, PSM+eDRX. 3 modes: Standalone (GSM 200kHz), In-band (inside LTE PRB), Guard Band (between LTE/WCDMA).
  • Modbus: 1979, Modicon. Master-slave. Request-Response. RTU (binary) / ASCII (text) / TCP (Ethernet). RS-232/RS-485 or Ethernet. Register model: Coils(1-bit), Discrete inputs, Holding registers, Input registers.
  • UART/SPI/I2C: UART=2-wire (TX-RX crossed), async, 115200bps, full-duplex, 1:1. SPI=4-wire (MOSI/MISO/SCLK/SS), sync, full-duplex, 8Mbps+, one SS per slave. I2C=2-wire (SDA/SCL+pullups), sync, half-duplex, 400kbps, unique 7/10-bit address, multi-device same bus.
  • Xively: PaaS (formerly Cosm/Pachube, Google). 4 concepts: Data Stream + Data Points + Triggers + Feed. Pull=auto HTTP server; Push=manual HTTP client. REST/MQTT/WebSockets. JSON/XML/CSV.
  • Virtualization types: Storage (abstract interface hides multi-server, eg iCloud), Network/NFV (virtual routers/firewalls on commodity hw), Server (multiple VMs on 1 server, hypervisor, eg VMware/KVM), Desktop (remote virtual desktops, eg Citrix).
  • 5 FCs of security: FC1=IdM, FC2=Authentication (MD5/SHA hash, only hash transmitted), FC3=Authorisation (ACL/RBAC/ABAC), FC4=Key Mgmt (gen/distrib/store/revoke), FC5=Trust & Reputation (dynamic trust scores).
  • Layered attacker: L1=Eavesdrop/Jam/Tamper→AES-CCM 128. L2=MAC spoof/ARP poison→ARP inspection. L3=IP spoof/RPL DIO forge→firewall L3-L4. L4=Port scan/SYN flood→DTLS. L5/6=XSS/SQLi→HTTPS/S-HTTP. CISCO: RBAC all layers.

Cloud Computing — Definition & Paradigm ★★★ Every Paper

Definition: Computing using a collection of Internet-based services delivered on a cloud provider's infrastructure. Like electricity — user doesn't need to know the source or infrastructure.

Data Collection Methods (low → high scale):

  • Device web server → Local files (SD card, HDD)
  • Coordinating node → Local distributed DBMS
  • Internet-connected server / data center
  • Cloud infrastructure — highest level, elastic and scalable

IoT/M2M devices → Internet firewall → Cloud (Collect + Store + Compute + Analyse)

Virtualization in Cloud ★★

Definition: Enables applications to run in independent, isolated environments on shared physical hardware.

Types of Virtualization:

  • Storage Virtualization — Abstract interface hides multi-server storage. User sees one storage entity. eg: Apple iCloud, Google Drive
  • Network Virtualization (NFV) — Multiple networks appear as one; virtual routers, firewalls on commodity servers
  • Server Virtualization — Multiple VMs on one physical server (hypervisors). Load balancing, isolation. eg: VMware, KVM, Hyper-V
  • Desktop Virtualization — Multiple virtual desktops on one OS. Remote access. eg: AWS WorkSpaces, Citrix

Xively Platform ★★★ Every Paper

Type: PaaS | Formerly Cosm / Pachube | Now owned by Google | First public cloud for commercial IoT | Open-source for Arduino/RPi

Key Concepts:

  • Data Stream — Continuous sensed data flow over Internet from a sensor
  • Data Points — Individual values at a specific time (free account: 10 feeds, stored 3 months)
  • Triggers — Actions on state change (eg: light level below threshold → streetlight ON)
  • Feed — Set of data points, data streams, messages, or alerts

Data Capture Modes:

  • Pull Method (Automatic Feed) — Data automatically collected from HTTP server
  • Push Method (Manual Feed) — Data written to Xively using an HTTP client

Features:

  • Protocols: REST, MQTT, WebSockets | Formats: JSON, XML, CSV
  • SDKs: Arduino, Java, Python, Ruby, Android, ARM mBed
  • Real-time data visualization, graphical plots, alert generation, historical data access

Features:

IoT Device Building Blocks ★★

IoT device = physical object embedded with sensors, actuators & communication that connects to Internet.

4 Building Blocks:

  • Sensing (Sensors) — On-board or external. Collects temperature, humidity, light, pressure data. eg: DHT11, LDR, pressure sensors
  • Actuation (Actuators) — Takes physical actions on commands. eg: relay switches, servo motors, LEDs, buzzers, valves
  • Communication Module — Sends/receives data. eg: Wi-Fi (802.11), BLE, ZigBee (802.15.4), LoRaWAN, NB-IoT, Cellular
  • Processing & Analysis — Makes sense of data locally. eg: CPU/MCU, RAM, Flash, on-chip ADC/DAC, SD card

SBC (Single Board Computer) like Raspberry Pi integrates all 4 blocks on one board.

GPIO Programming — LED & Push Button (Python) ★★★ Every Paper

LED Blink on GPIO 20:

import RPi.GPIO as GPIO import time GPIO.setmode(GPIO.BCM) # BCM Broadcom pin numbering GPIO.setup(20, GPIO.OUT) # GPIO 20 = LED output while True: GPIO.output(20, GPIO.HIGH) # LED ON (3.3V) time.sleep(1) GPIO.output(20, GPIO.LOW) # LED OFF (0V) time.sleep(1) GPIO.cleanup() # Reset all pins on exit

Switch (GPIO 21) controls LED (GPIO 20):

GPIO.setup(21, GPIO.IN, pull_up_down=GPIO.PUD_UP) # Button = input + pull-up GPIO.setup(20, GPIO.OUT) # LED = output try: while True: if GPIO.input(21) == False: # Button pressed → GPIO reads LOW GPIO.output(20, GPIO.HIGH) # LED ON else: GPIO.output(20, GPIO.LOW) # LED OFF time.sleep(0.1) finally: GPIO.cleanup()

Circuit:

  • LED: GPIO20 → 330Ω resistor → LED Anode(+) → LED Cathode(−) → GND
  • Button: GPIO21 ↔ Push button ↔ GND | Pull-up enabled in software (reads HIGH when open, LOW when pressed)

Run with: sudo python led.py | GPIO.BCM = Broadcom numbering | GPIO.BOARD = physical pin numbering

Message Integrity & Non-Repudiation ★★★

Message Integrity: Ensures data is unaltered during transmission/storage. Decrypted data must be identical to original.

How it works (Hash/Digest method):

  • Step 1: Sender computes h0 = Hash(M0, K) using secret key K
  • Step 2: h0 appended to message and sent
  • Step 3: Receiver computes h1 = Hash(M1, K)
  • Step 4: Compare — if h1 = h0 → PASS (message unchanged); if h1 ≠ h0 → FAIL (tampered)
Hash Algorithms: MD5 → 128-bit digest | SHA-256 → 256-bit digest

Non-Repudiation: Digital signature certified by Trusted Third Party (TTP). Sender cannot later deny having sent the message.

Maps attacks to specific OSI layers. Enables defence-in-depth security design for IoT.

  • L1–6: Role-based security | L1–4: Anti-tamper detection | L1–6: Data protection & confidentiality | L1–6: IP protection

Smart Street Lighting (IoT Use Case) ★★★ Every Paper

Why LED? Less energy, longer lifespan, adjustable colour/intensity, responds instantly to digital control

Architecture (top → bottom):

  • Cloud Application Layer — Scheduling, dimming control, fault alerts, historical analytics, energy reports
  • City Network/Gateway Layer — Aggregation switches, protocol gateways
  • Lighting Node Layer — LED + Controller + Light Sensor + Driver. Monitors status, reports faults, receives dim/bright commands from cloud

Protocols:

ZigBee, Wi-Fi, 802.15.4g (Wi-SUN), LoRaWAN, Cellular

Key Features:

  • Remote management via cloud app
  • Automated scheduling (time of day, season, weather, city zone)
  • Adaptive dimming/brightening based on ambient light sensors
  • Fault monitoring — lamp failure detected immediately, maintenance dispatched
  • Energy savings through precise control; no unnecessary burning during daylight

Bonus services on same lamp post: Air quality sensors, traffic cameras, public Wi-Fi access points — sharing existing power & comms infrastructure

M4: Raspberry Pi Interfaces — UART, SPI, I2C ★★★ Every Paper

UART = 2-wire (TX-RX crossed), async, 115200 bps, full-duplex, 1:1. SPI = 4-wire (MOSI/MISO/SCLK/SS), sync, full-duplex, 8 Mbps+, one SS per slave. I2C = 2-wire (SDA/SCL + pull-ups), sync, half-duplex, 400 kbps, multiple slaves on same bus with unique 7-bit/10-bit addresses. Memory hook: UART = simple/slow; SPI = fast/more wires; I2C = 2-wire/multi-device.

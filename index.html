IoT Definition ★★★ Every Paper

Definition 1 — Based on Technology:

IoT is a new revolution to the internet due to advancement in sensor networks, mobile devices, wireless communication, networking and cloud technologies.

Definition 2 — Based on Infrastructure:

IoT is a dynamic global network infrastructure of physical and virtual objects having unique identities, embedded with software, sensors, actuators, electronics and network connectivity to facilitate intelligent applications by collecting and exchanging data.

Goal of IoT: Connect non-internet devices (thermostats, utility meters, irrigation pumps, sensors) to the internet to make energy, logistics, agriculture, and industrial control domains smarter.

IoT Characteristics ★★★ ×6 Every Paper

5 Characteristics:

  • Dynamic & Self-Adapting (1 mark): Adapts to changing contexts and takes actions accordingly. Example: Surveillance camera adjusts recording quality based on motion detected; switches to IR mode at night.
  • Self-Configuring (½): Devices upgrade software and set up networking with minimal user intervention. New device can be added to existing network automatically. Example: Smart device connecting over free Wi-Fi access.
  • Interoperable Communication Protocols (½): Allows different devices (different architectures) to communicate with each other and different networks. Example: Mi Phone controls smart AC and smart TV from different manufacturers.
  • Unique Identity (½): Every device has a unique identity (IP address/URI) for identification throughout the network. Intelligent interfaces allow remote query, monitoring, and control.
  • Integrated into Information Network (½): Devices are connected to the network to share information. Collective intelligence makes them smarter. Example: Weather monitoring where data from multiple sensor nodes is aggregated to predict weather.

REST vs WebSocket APIs ★★★ ×4

IoT Enabling Technologies ★★★ ×4

5 Enabling Technologies:

  • WSN (Wireless Sensor Networks): Distributed devices + wireless sensors monitoring environment. End nodes → routers → coordinator → internet. Protocols: IEEE 802.15.4, ZigBee (2.4 GHz, 250 kbps, 10–100 m). Examples: weather monitoring, indoor air quality, smart grids.
  • Cloud Computing: On-demand network access to computing resources; pay-as-you-go. IaaS (hire VMs/storage), PaaS (develop+deploy apps on cloud), SaaS (complete software via browser).
  • Big Data Analytics — 5 Vs: Volume (massive data, no threshold), Velocity (rate of generation), Variety (text, audio, video; structured/semi/unstructured), Veracity (data in doubt/errors), Value (actionable insights).
  • Embedded Systems: Hardware + software for specific task. Components: MCU/microprocessor, memory, networking (Ethernet, WiFi), I/O, flash storage. Uses RTOS. Examples: digital watch, camera, vending machines.
  • Communication Protocols: Backbone of IoT — defines data exchange format, encoding, addressing, routing, flow control, retransmission of lost packets; enables interoperability.

DIK Principle + OSI Model ★★ ×2

DIK Principle (Data → Information → Knowledge):

  • Data: Raw, unprocessed values from IoT devices — no meaning until contextualised
  • Information: Raw data processed, contextualised, filtered, categorised, condensed — algorithms extract information
  • Knowledge: Inferred from information by organising/structuring — put into action for specific objectives

Example: Raw sensor ((72,45);(84,56)) → add context (temp & humidity/min) → Information → compute 5-min average → refined info → alert if avg temp >120°F → Knowledge in action.

IoT Architecture (4-Layer) + Message Integrity ★★★ ×3

Application Layer ← Domain apps (smart home, health, industry) Middleware Layer ← Data processing, device mgmt, analytics Network Layer ← IP routing, gateways, WiFi/LoRa/Zigbee Perception Layer ← Physical sensors, actuators, RFID

Message Integrity:

Definition: Message remains unaltered during communication. Decrypted data = original pre-encryption data.

  • Sender: Hash(Message M0 + Key K) → hash value h0; transmit (M0 + h0)
  • Receiver: Hash(received M1 + Key K) → h1; compare h1 vs h0
  • h1 = h0 → Integrity OK ✓ | h1 ≠ h0 → Tampered! ✗

Hash properties: One-way (cannot reverse-compute), Deterministic (same input → same hash), Avalanche effect (small change → completely different hash), Collision-resistant

Algorithms: MD5 (128-bit), SHA-256 (256-bit)

IoT vs M2M ★★★ ×6 Every Paper

M2M Definition: Networking of machines for remote monitoring, control, and data exchange. M2M area networks comprise machines with embedded hardware modules for sensing, actuation, and communication.

M2M Architecture: [M2M AREA NETWORKS] → [M2M GATEWAY] → [M2M CORE NETWORK] → [M2M APPLICATIONS] (Machines: ZigBee, (Protocol (Wired/Wireless IP) (App Server + BT, ModBus, Z-Wave) Translation, Database) Virtual Nodes, IP Routing)

M2M Gateway functions:

  • Protocol Translation: Non-IP (ZigBee, Bluetooth, ModBus, Z-Wave) → IP-based protocols
  • Virtual Node Management: Each M2M node represented as virtual node with native protocol handler + proxy
  • IP Routing: Routes translated data to M2M core network via standard IP routing
  • Data Aggregation + Security: Acts as security boundary between local area network and internet

SDN — Issues & 3-Layer Architecture ★★★ ×5

Definition: SDN separates the control plane from the data plane and centralises network control in a software-based controller, enabling programmable, flexible, and scalable networks.

Issues with Conventional Networks:

  • Complex devices: Routers/switches must implement many distributed protocols (OSPF, BGP, STP, VLAN, QoS) simultaneously — hard to change dynamically
  • Coupled control + data planes: Tightly coupled on same hardware — limits flexibility; routing decisions + forwarding both on same device
  • Management overhead: Each device managed independently from multiple vendors — no unified network view; complex troubleshooting
  • Limited scalability: IoT requires distributed algorithms on large numbers of VMs — impossible with conventional approaches
SDN 3-Layer Architecture: +--------------------------------------------------+ | APPLICATION LAYER | | [Network App] [Firewall] [Load Balancer] | +--------------------------------------------------+ ↕ Northbound Open API (REST/JSON) +--------------------------------------------------+ | CONTROL LAYER | | +-----------------------------------------+ | | | SDN Controller (Network Operating Sys) | | | +-----------------------------------------+ | +--------------------------------------------------+ ↕ Southbound Open API (OpenFlow) +--------------------------------------------------+ | INFRASTRUCTURE LAYER | | [OpenFlow SW] [OpenFlow SW] [OpenFlow SW] | +--------------------------------------------------+

3 SDN Components:

ComponentDescription
Application LayerBusiness-level apps (firewall, load balancer, QoS) interact with controller via Northbound API (REST/JSON)
SDN Controller (NOS)Centralised brain; decouples control from data plane; rapid reconfiguration from single point; communicates via southbound API
Infrastructure LayerOpenFlow-capable switches — simple packet forwarding only; flow tables programmed by controller (match fields + counters + actions)

APIs:

  • Northbound (REST/JSON): Applications → Controller; network service requests and policies
  • Southbound (OpenFlow, ONF standard): Controller → Switches; installs forwarding rules in flow tables
  • East-West: Controller ↔ Controller; for distributed SDN deployments

SDN for IoT: Dynamic bandwidth allocation for diverse IoT traffic; multi-tenant virtual networks; network slicing (different virtual networks for different IoT apps on same physical infrastructure); centralised security policy enforcement; scales as IoT device count changes.

WSN — Constraints of Smart Objects ★★★ ×5

WSN Definition: A Wireless Sensor Network is a network of wirelessly connected smart objects (nodes) that sense, measure, and communicate information about their environment. SANET = Sensor/Actuator Network (nodes sense + actuate). Smart objects in WSN = 'nodes' = 'constrained nodes'.

WSN Multi-hop Topology: [Sensor Node 1]--+ [Sensor Node 2]--+--[mesh]--[Sensor Node 4]--[Gateway/Base Station]--Internet [Sensor Node 3]--+

5 Constraints / Limitations of Smart Objects in WSNs:

ConstraintDetailsImpact
Limited Processing PowerLow-power 8–32 bit MCUs; cannot do complex computations locallyData must be forwarded to gateway/cloud; aggregation must be lightweight
Limited MemoryOnly a few KB of RAM (e.g. 4–10 KB RAM, ~48 KB flash)No large data buffers; no complex protocol stacks; no large datasets

Impact on WSN Design (solutions):

  • Large-scale node deployment to compensate for individual limitations
  • Hierarchical topologies — cluster heads aggregate data from nearby nodes
  • Data aggregation — compute average of 6 nodes and send ONE value instead of six
  • Event-driven or periodic communication patterns conserve energy
  • Lightweight protocols: 6LoWPAN (IPv6 over 802.15.4), CoAP (constrained application), RPL (routing for LLNs)
  • Redundant deployment — individual sensors may be inaccurate; redundancy improves accuracy

WSN Communication Patterns:

  • Event-driven: Triggered only when object detects a particular event/threshold — e.g. fire sensor sends alarm only when temp > 80°C
  • Periodic: Fixed time intervals — e.g. weather station reports every 5 minutes regardless

LoRaWAN Architecture ★★★ ×6 Top Priority!

Definition: LPWAN protocol built on LoRa modulation for long-range IoT with low power. Maintained by LoRa Alliance. LoRa modulation = proprietary to Semtech.

Star-of-Stars Topology: [End Devices] --> [Gateways] --> [Network Server] --> [Application Server] (Class A/B/C) (bridges) (de-dup, ADR) (processes + API)

Device Classes:

Physical Layer (LoRa CSS Modulation):

  • CSS = Chirp Spread Spectrum — unique to LoRa
  • SF7–SF12: Higher SF = longer range, lower data rate
  • Frequencies: 865–867 MHz (India), 863–870 MHz (Europe), 902–928 MHz (US)
  • Data rates: 0.3–50 kbps | Range: 2–5 km urban, 15+ km rural

MAC Layer:

  • Payload: MAC Header (1B) + MAC Payload + MIC (4B)
  • DevEUI = 64-bit globally unique ID | DevAddr = 32-bit network address
  • Security: AES-128; NwkSKey (MAC layer) + AppSKey (payload)
  • ADR: Network server dynamically adjusts SF and TX power

RPL — Routing Protocol for LLNs ★★★ ×4

Definition: IPv6 routing protocol for Low-Power Lossy Networks. IETF RFC 6550. Builds a DODAG (Destination-Oriented DAG) rooted at border router.

[Root/Border Router] Rank 0 / \ [Node A] [Node B] Rank 1 / \ \ [Node C][Node D] [Node E] Rank 2

How RPL works:

  • DIO (DODAG Information Object) — sent downward; nodes compute rank and join
  • DAO (Destination Advertisement Object) — sent upward; installs downward routes
  • DIS (DODAG Info Solicitation) — nodes discover DODAG

Objective Functions: OF0 (minimize hops) | MRHOF (minimize ETX — most reliable links)

Features: Self-healing, multi-topology, Trickle timer (reduces overhead in stable networks)

6LoWPAN Adaptation Layer ★★★ ×3

Problem: IPv6 min MTU = 1280 bytes, but 802.15.4 max frame = 127 bytes. 6LoWPAN bridges this gap.

Stack position: Application(CoAP) → UDP → IPv6 → [6LoWPAN] → IEEE 802.15.4

4 Key Functions:

  • Header Compression (IPHC): IPv6 header 40B → 2–3B; UDP 8B → 4B
  • Fragmentation & Reassembly: Large IPv6 packets split into multiple 802.15.4 frames
  • Mesh Addressing: L2 routing header for multi-hop 802.15.4 networks
  • Auto-configuration: SLAAC using 802.15.4 MAC address as EUI-64

ZigBee Protocol Architecture ★★★ ×3

Virtualization — Data Stores, Networks, Servers ★★ ×2 | 6 Marks

Definition: Creates abstract, logical representation of physical IT resources — partitioned, isolated, shared among multiple users. Each virtual resource behaves as if it were a dedicated physical resource.

Core concept: Apps execute in isolated virtual environments; need not be aware of physical platform — only Internet connectivity required. Enables multi-tenancy.

Physical: [ Servers ] [ Storage Arrays ] [ Network Switches ] | | | Virt. Layer: [Hypervisor] [SAN/NAS Abstraction] [SDN/NFV] | | | Virtual: [ VMs ] [Logical Drives] [Virtual Networks] | Application: [IoT Apps / Cloud Services / User Apps]

1. Storage Virtualization (2 marks):

  • Abstracts multiple physical disks into a single logical storage pool
  • App accesses storage via: database interface, file system, logical drive, or disk drive
  • Physical layout is transparent to app — may span multiple servers or data centers
  • Software (SAN, NAS, RAID) aggregates physical disks into logical volumes
  • IoT: Sensor data stored across distributed cloud nodes; app accesses via single API (e.g., AWS S3)

2. Network Virtualization / NFV (2 marks):

  • NFV = decouples network functions (firewall, load balancer, NAT, IDS) from dedicated hardware → runs as software on COTS servers
  • Network functions run as VNFs (Virtual Network Functions) on general-purpose servers
  • Eliminates expensive proprietary appliances; enables rapid provisioning and scaling
  • SDN complements NFV: separates control plane from data plane
  • IoT: Virtual firewalls and load balancers protect/scale IoT gateways dynamically

3. Server Virtualization (2 marks):

  • Multiple VMs run on single physical server; each has own OS + apps — fully isolated
  • Hypervisor (Type 1: bare-metal / Type 2: hosted) manages and allocates resources among VMs
  • One physical server can host 10–100+ VMs; crash of one VM does not affect others
  • Snapshot and live migration: VMs can be paused, backed up, moved without downtime
  • IoT: IoT cloud platforms run device management, analytics, APIs as separate VMs
